## 1. Topology

Cloud SQL Postgres 15, single regional pair on the caller's VPC:

- **Primary** (`<name_prefix>-pg-primary`) — writes + reads, private IP only, peered via Service Networking. Backups + PITR on (7-day retention, 03:00 start), Query Insights on, CMEK via `kms_key_id`, `ssl_mode = ENCRYPTED_ONLY`, ENTERPRISE edition, PD_SSD with disk autoresize.
- **Read replica** (`<name_prefix>-pg-replica`) — same region, same VPC, `failover_target = false`. Apps that want read scaling consume `DATABASE_URL_READ` (output: `cloudsql-replica-connection-name`).
- **Single `eqp` database**, **single `eqp` Postgres superuser** (32-char `random_password.special = false`, so it survives shell quoting in `DATABASE_URL`). Per-service Prisma migrations create their own SCHEMAS inside `eqp` (e.g. `iam`, `transaction`, `payment_gateway`) — multi-schema design, one Postgres user spans all.

Consumed today by **transaction-service** + **payment-gateway-service** (cloud-sql-proxy sidecar on each). Other 9 NestJS services are still on Firestore mid-migration.

## 2. Provisioning (`infra/terraform/modules/cloud-sql/main.tf`)

- Random password generated by `random_password.postgres` (length 32, special chars excluded).
- Persisted to **3 Secret Manager secrets**: `cloudsql-postgres-password`, `cloudsql-connection-name`, `cloudsql-replica-connection-name`.
- Outputs: `primary_connection_name`, `replica_connection_name`, `primary_private_ip`, `replica_private_ip`, `database_name`, `user_name`, `postgres_password` (sensitive — captured to `~/Desktop/keys-db` on first apply).
- IAM bindings live **outside** the module (separation of concerns): `roles/cloudsql.client` per-consumer GSA via `module.workload_identity.workload_definitions`; `roles/secretmanager.secretAccessor` on the 3 secrets via `module.secret_iam.secret_grants`.
- `cloudsql.iam_authentication = on` is already set as a `database_flags` entry — the instance is prepped for IAM auth, app-side wiring is the follow-up (see §8).
- VPC peering: only ONE `google_service_networking_connection` per (VPC, service); Cloud SQL's `/20` `google_compute_global_address` rides along on `module.cloudbuild_pool`'s connection via the module's `additional_peering_ranges` arg. `module.cloud_sql.depends_on = [module.cloudbuild_pool]` so the peering reconciles BEFORE Cloud SQL allocates.
- Dev: password auth in use today (PR #119, applied on cips-test 2026-04-30). Password rotated 2026-05-01 after a `set -x` leak.

## 3. Prisma schema (`packages/database/prisma/schema.prisma`)

- Provider `postgresql`, `DATABASE_URL` env var.
- Models: `User`, `Business`, `Transaction`, `Card`, `Session`, `AuditLog` (plus `ApiKey` added in `20260505120000_add_api_key_model`).
- Money columns are **`Decimal @db.Decimal(18, 4)`** (e.g. `Transaction.amount`). Never `Float`.
- Currency stored as `String @db.VarChar(3)` (ISO-4217).
- snake_case at the DB, camelCase in TS — every field uses `@map("col_name")`, every model uses `@@map("table_name")`. Example: `firebaseUid String @unique @map("firebase_uid")`, `@@map("users")`.
- Indexes for hot paths: `@@index([businessId, status])` on `Transaction`, `@@index([userId])` on `Session`, `@@index([userId, action])` + `@@index([entity, entityId])` on `AuditLog`, `@@index([prefix, status])` on `ApiKey`.
- Enums (`Role`, `BusinessStatus`, `TransactionStatus`, `TransactionType`, `CardStatus`, `ApiKeyEnv`, `ApiKeyStatus`) are Prisma enums → Postgres native `CREATE TYPE`.

Migrations live under `packages/database/prisma/migrations/<timestamp>_<name>/migration.sql`. Lock file pins provider to `postgresql`.

## 4. Migrations

- Generated locally via `bun run --cwd packages/database db:migrate:dev`.
- Initial migration `20260501141200_init` baseline was hand-written + reviewed (PR #123). Subsequent migrations: `20260505120000_add_api_key_model`, `20260505140000_drop_api_key_audit_events`, `20260505160000_allow_business_owner_id_null`.
- `db:migrate:deploy` is what runs in CI/prod (not currently wired — migrator runs ad-hoc via Cloud Build per §5).

## 5. Migrator job — Cloud Build private worker pool (`infra/cloudbuild-migrate.yaml`)

Why NOT local `cloud-sql-proxy`: instance is private-IP only; `--private-ip` from a laptop hits "Couldn't find route" (no VPN).

Why NOT in-cluster K8s Job: Binary Authorization blocks third-party images. `postgres:15-bookworm` is not attested, the image admission webhook denies pod creation.

Why Cloud Build works: the private worker pool (`eqp-<env>-cb-pool`) is peered with the VPC via Service Networking, so it can reach Cloud SQL's private IP directly. The runner uses `gcr.io/google.com/cloudsdktool/cloud-sdk:slim` — already on the BinAuthz whitelist.

Why raw `psql + generated SQL` (vs `prisma migrate deploy`): worker doesn't have node + bun + the full repo. Installing inflates build + image trust. Generating SQL via `prisma migrate diff` ahead of time keeps the build minimal. **Tradeoff:** `_prisma_migrations` table doesn't get populated — see Trap 2.

## 6. Submit pattern

```bash
# 1. Generate migration SQL locally (no DB needed):
cd packages/database
bun run prisma migrate diff --from-empty --to-schema-datamodel ./prisma/schema.prisma --script > /tmp/init.sql

# 2. Stage:
cp /tmp/init.sql /tmp/migrate-src/init.sql
cp infra/cloudbuild-migrate.yaml /tmp/migrate-src/cloudbuild.yaml

# 3. Submit:
cd /tmp/migrate-src
gcloud builds submit \
  --config=cloudbuild.yaml \
  --service-account=projects/<project>/serviceAccounts/eqp-<env>-cloudbuild@<project>.iam.gserviceaccount.com \
  --project=<project> --region=us-central1 .
```

Use the **named** `eqp-<env>-cloudbuild` SA, not the default Cloud Build SA. Required IAM on that SA:

- `roles/cloudsql.client` on the project
- `roles/secretmanager.secretAccessor` on `cloudsql-postgres-password`

Codify per env in `local.workload_definitions` + `local.secret_grants` (`infra/terraform/envs/<env>/main.tf`).

## 7. Traps

1. **No `set -x`.** xtrace echoes the password on the line that exports `PGPASSWORD` via `$(gcloud secrets versions access ...)`, leaking it to the build log. Use `set -euo pipefail` only. If leaked: rotate via `terraform apply -replace=module.cloud_sql.random_password.postgres`, then re-fetch from Secret Manager.
2. **Raw `psql -f migration.sql` doesn't populate `_prisma_migrations`.** Subsequent `prisma migrate deploy` runs can't reason about state. After the first apply, run `bun run --cwd packages/database prisma migrate resolve --applied <name>` against the instance once.
3. **Cloud Build YAML `$VAR` substitution.** Cloud Build templates `$VAR` at build time; escape shell vars as `$$VAR` (e.g. `$$(gcloud secrets versions access ...)`, `$$TAG`, `$$ENV`).
4. **`cloud-sql-proxy` startup race in pods.** The proxy sidecar may not be ready before app container hits `pg_isready`. Cloud Deploy verify-job timeouts on tx + payment-gateway are a recurring flake — see Plan 3c item 7. Fix path: tune cloud-sql-proxy `startupProbe`.
5. **First-deploy SPC deadlock.** cloud-sql-proxy + app reference K8s Secrets `tx-cloudsql-*` / `pg-cloudsql-*` via `valueFrom.secretKeyRef`. Those are created by the SPC's `secretObjects` block only AFTER volume mount. Unlike the portal, tx + payment can't use `optional: true` (empty password = startup crash). Operator must bootstrap the K8s Secrets via one-shot `kubectl create secret generic` BEFORE first rollout.

## 8. IAM auth follow-up (before staging/prod cutover)

The instance is already prepped (`cloudsql.iam_authentication = on`). To enable:

1. **Terraform** — add `google_sql_user` resources with `type = "CLOUD_IAM_SERVICE_ACCOUNT"` for each consumer GSA (transaction-service, payment-gateway-service, future Postgres consumers).
2. **Terraform** — add `roles/cloudsql.instanceUser` on the project for each consumer GSA (extend `local.workload_definitions` in `infra/terraform/envs/<env>/main.tf`).
3. **K8s** — in `infra/k8s/base/{transaction,payment-gateway}-service/deployment.yaml`:
   - `cloud-sql-proxy` args += `"--auto-iam-authn"`
   - `DATABASE_URL` drops `:$(POSTGRES_PASSWORD)`, becomes `postgresql://<gsa-prefix>@127.0.0.1:5432/eqp`
   - Drop the `POSTGRES_PASSWORD` env var
4. **PCI 8.x for staging/prod** — add `google_sql_user` of `type = "CLOUD_IAM_GROUP"` keyed off `var.dba_group_email` (e.g. `eqp-db-admins@cloud-invest.com`). Group membership + MFA at Workspace level handles human DBA auth.

Keep `random_password` + `cloudsql-postgres-password` secret as a break-glass DBA credential — PCI 8.6 permits this as long as the human-account path is MFA'd via the group above.

Deferred 2026-04-30 to validate password path first.

## 9. `db:generate` before `check-types`

`bun run --cwd packages/database db:generate` MUST run before `check-types` (and `test`, `build`, `test:coverage`) in any CI job that touches `@repo/database` consumers.

Root cause: Cloud Build / GitHub Actions runners install with `bun install --frozen-lockfile --ignore-scripts`, suppressing the `postinstall` hook that would normally regenerate the Prisma client. Without an explicit `prisma generate`, `node_modules/.prisma/client/default` doesn't exist and any consumer fails to compile with `TS2305: Module '"@prisma/client"' has no exported member 'PrismaClient'`. Fixed in PR #150.

For the runtime image, `docker/Dockerfile.nestjs` runner stage also runs `bun install --ignore-scripts` and must call `bunx prisma generate --schema ./packages/database/prisma/schema.prisma` (guarded with `if [ -f ./packages/database/prisma/schema.prisma ]; then ... fi` because `turbo prune --docker` excludes `packages/database/` for services that don't depend on Prisma — PRs #158, #160).

`.github/workflows/ci.yml` reflects this: every job (`validate`, `test`, `test-coverage`, `build`) runs `bun run --cwd packages/database db:generate` immediately after `bun install --frozen-lockfile`.
